Actor-Critic Methods: Two Heads, One Goal

How RL split the problem of "what to do" from "how good is this state"

Start With What You Already Know

In your optimal control course, you encountered the value function V(x) — the expected total future cost starting from state x under the optimal policy. And you encountered the policy π(x) — the actual control law that tells you what input to apply at state x. In RL these are still the same two objects. Actor-Critic is simply: learn both of them at the same time, using each to help the other.

The Three Families of RL Algorithms

Before Actor-Critic, there were two extreme approaches:

Family What it learns How it acts Problem
Value-based (DQN) Q(s,a) for all actions argmax over Q Doesn't work with continuous actions; can't represent stochastic policies
Policy Gradient (REINFORCE) π(a|s) directly Sample from π Very high variance in gradient estimates; slow to learn
Actor-Critic Both π(a|s) and V(s) Sample from π; update using V More complex; two things to tune — but solves both above problems

The Core Idea: Policy Gradient Needs a Baseline

A pure policy gradient method (REINFORCE) updates the policy using the signal:

∇θ J(θ) = E[ ∇θ log π(a|s) · G ] G = total discounted return from this episode. This is the REINFORCE gradient.

The problem: G has massive variance. If one episode got return 480 and another got 502, the gradient estimate is swamped by noise. You need thousands of episodes to get a reliable signal.

The fix: subtract a baseline b(s) from the return. The gradient is still unbiased (the baseline can be anything that doesn't depend on the action), but the variance drops dramatically.

∇θ J(θ) = E[ ∇θ log π(a|s) · (G − b(s)) ] Subtracting baseline b(s) reduces variance without introducing bias.

The best possible baseline? The value function V(s). The quantity (G - V(s)) even has a name:

A(s, a) = Q(s, a) − V(s) The Advantage Function: "how much better is action a than the average action in state s?"
Connecting to optimal control: The advantage function is essentially the difference between the Q-function (cost-to-go with specific action a) and the V-function (optimal cost-to-go). In LQR terms, it measures how suboptimal a given control input is relative to the value function.

The Two-Head Architecture

So Actor-Critic uses two function approximators (usually neural networks):

┌─────────────────────────────────────────────────────┐ │ NEURAL NETWORK │ │ │ │ Input: state s = [x, ẋ, θ, θ̇] │ │ (for CartPole, a 4-dim vector) │ │ │ │ │ [shared or separate hidden layers] │ │ │ │ │ │ ┌────────▼───────┐ ┌───────▼────────┐ │ │ │ ACTOR HEAD │ │ CRITIC HEAD │ │ │ │ π(a | s; θ) │ │ V(s; φ) │ │ │ │ │ │ │ │ │ │ Outputs action │ │ Outputs a │ │ │ │ probabilities │ │ scalar: "how │ │ │ │ [P(left), │ │ good is this │ │ │ │ P(right)] │ │ state?" │ │ │ └────────────────┘ └────────────────┘ │ └─────────────────────────────────────────────────────┘ ACTOR = the policy. Decides what to do. CRITIC = the value function. Judges how good the situation is.

The Training Loop (in words)

Here is one full update cycle of a basic Actor-Critic:

1. Actor observes state s_t
2. Actor samples action a_t ~ π(·|s_t)
3. Environment returns r_t and s_{t+1}

4. Critic computes TD error (the "surprise"):
      δ_t = r_t + γ·V(s_{t+1}) − V(s_t)
           └── what actually happened ──┘   └── what was expected ──┘

5. Critic updates: minimize δ_t² 
   (make V(s_t) better predict future rewards)

6. Actor updates: push probability of a_t UP if δ_t > 0
                                         DOWN if δ_t < 0
   gradient = ∇θ log π(a_t|s_t) · δ_t
The TD error δ_t IS the advantage estimate. It says: "I expected V(s_t) worth of future reward. I got r_t + γV(s_{t+1}). The difference is how much better/worse than expected this action was." This is a one-step approximation to the true advantage A(s,a).

Why "Critic" Helps the "Actor"

Without the critic, REINFORCE waits until the end of the episode to compute G. This means: (1) you must wait for episode termination, (2) you average over the entire trajectory, (3) variance is sky-high.

With the critic, you get a signal after every single step. The critic's V(s) is a learned "prediction" of future reward that substitutes for waiting for G. This is the same reasoning behind temporal difference (TD) methods in RL — bootstrap from your own estimate rather than waiting for ground truth.

A3C and A2C: Going Parallel

In 2016, DeepMind proposed A3C (Asynchronous Advantage Actor-Critic). The idea: run N copies of the environment in parallel with N actors. Each actor independently collects experience and computes gradients, then asynchronously pushes updates to a shared network. A2C (the synchronous version, which PPO builds on) waits for all N actors to finish before updating — cleaner gradients, easier to reason about.

This is the family PPO belongs to. It is an Actor-Critic method. The actor is the policy network, the critic is the value network. What PPO adds on top — the clipping trick — is what the next blog is about.

Summary: Actor = policy π(a|s), learns what action to take. Critic = value function V(s), learns how good a state is and provides a low-variance training signal for the actor. Together they solve what neither can do well alone.

Proximal Policy Optimization: Reading the Paper

Schulman et al., OpenAI 2017 — decoded from first principles

The Problem PPO Is Solving

Vanilla policy gradient has a dangerous failure mode. You compute a gradient, take a big step, and the policy changes so much that the new policy collects completely different data. The next gradient estimate is now computed from data that's irrelevant to the current policy. Training destabilizes, performance collapses. In practice you have to use tiny learning rates — which means painfully slow learning.

The question PPO answers: how large can the policy update step be before it becomes destructive? And can we enforce a safe step size without complex second-order optimization?

The Predecessor: TRPO

Trust Region Policy Optimization (also from Schulman, 2015) solved this with a hard constraint. At each update, maximize the expected advantage, but don't let the new policy deviate too far from the old one, measured by KL divergence:

maximize E[ (π_new(a|s) / π_old(a|s)) · Â ] subject to E[ KL(π_old, π_new) ] ≤ δ TRPO: maximize performance improvement while keeping the policy "close" to what it was.

TRPO works well but has two problems: (1) it requires computing second-order derivatives (conjugate gradient), which is expensive and hard to implement. (2) It doesn't work with architectures that share parameters between actor and critic, or use dropout. PPO keeps the spirit of TRPO but replaces the hard constraint with a clever objective function.

The PPO Objective: Clipping

Define the probability ratio:

r_t(θ) = π_θ(a_t | s_t) / π_θ_old(a_t | s_t) How much more (or less) likely is the current policy to take the same action as the old policy?

When r = 1, new and old policies are identical. When r = 1.5, the new policy is 50% more likely to take that action. When r = 0.5, the new policy is half as likely.

The naive surrogate objective (from TRPO) is just:

L_CPI(θ) = E[ r_t(θ) · Â_t ] Without any constraint, maximizing this causes destructively large policy updates.

PPO replaces this with a clipped version that refuses to give credit for going too far:

L_CLIP(θ) = E[ min( r_t · Â_t , clip(r_t, 1−ε, 1+ε) · Â_t ) ] The core PPO objective. ε is typically 0.2. This is equation (7) from the paper.

Understanding the Clip: Two Cases

The clip works differently depending on the sign of the advantage. Let's think through both:

Case 1: Â > 0 (good action — we want more of this)

We want to increase π(a|s), so r_t should grow above 1. But if r_t > 1+ε, the clip kicks in and returns 1+ε instead of r_t. The min then picks the clipped value.

Effect: We increase the probability of this action, but only up to 1+ε times what it was. We can't go further even if the gradient says to.
Case 2: Â < 0 (bad action — we want less of this)

We want to decrease π(a|s), so r_t should shrink below 1. But if r_t < 1−ε, the clip kicks in and returns 1−ε. The min then picks the unclipped value (which is more negative).

Effect: We still decrease the probability of this action fully. The clip doesn't protect bad actions.
The deep insight: The clip creates a "pessimistic lower bound." It says: "If the update would be beneficial and we're about to do a lot of it, stop early. If the update is harmful, let it through." The agent always errs on the side of caution when making large changes.

The Advantage Estimator: GAE

PPO uses Generalized Advantage Estimation (GAE) to compute Â_t. This is a key component. Recall the one-step TD error:

δ_t = r_t + γ·V(s_{t+1}) − V(s_t) One-step TD error: immediate reward + discounted future value estimate − current value estimate

GAE combines multiple steps of these TD errors with an exponential decay:

Â_t = δ_t + (γλ)·δ_{t+1} + (γλ)²·δ_{t+2} + ... + (γλ)^{T-t-1}·δ_{T-1} GAE(γ, λ). λ controls the bias-variance tradeoff. Paper uses λ=0.95, γ=0.99.

When λ=0: pure one-step TD (low variance, high bias — critic errors propagate). When λ=1: full Monte Carlo return (zero bias, high variance). λ=0.95 is a carefully chosen middle ground that works well empirically across many tasks.

The Full PPO Loss Function

The paper combines three terms into one loss that gets optimized:

L(θ) = E[ L_CLIP(θ) − c₁·L_VF(θ) + c₂·S[π](s) ] Equation (9) from the paper. Three terms: policy loss, value loss, entropy bonus.
Term What It Is Why It's There
L_CLIP The clipped surrogate policy loss Improves the policy while preventing large updates
−c₁·L_VF Value function MSE: (V(s) − V_target)² Trains the critic to predict returns accurately
+c₂·S[π] Entropy of the policy distribution Encourages exploration — prevents the policy from becoming too deterministic too early

The PPO Algorithm (from the paper)

Algorithm 1 from the paper, explained line by line:

for iteration = 1, 2, ... do

  # Phase 1: Data Collection
  for each of N parallel actors:
    run current policy π_θ_old for T timesteps
    compute advantage estimates Â_1 ... Â_T using GAE

  # Phase 2: Policy Update
  for K epochs:
    for each minibatch of size M from the NT timesteps:
      compute L_CLIP + L_VF + entropy
      one step of Adam gradient ascent
  
  θ_old ← θ   # old policy becomes the new policy for next round

end for
The key innovation: Standard policy gradient does ONE gradient step per batch of data. PPO does K epochs (typically 10) of minibatch updates on the same collected data. This is far more data-efficient. The clipping is what makes this safe — without it, multiple updates on the same data cause the policy to drift too far.

Hyperparameters from the Paper

Hyperparameter Value Used What It Controls
ε (clip ratio) 0.2 How much the policy can change per update. ±20% probability ratio allowed.
γ (discount) 0.99 How much future rewards are worth. 0.99 means reward 100 steps away is worth 0.99^100 ≈ 37% of immediate reward.
λ (GAE) 0.95 Bias-variance tradeoff in advantage estimation.
T (horizon) 2048 Steps collected per actor before each update.
K (epochs) 10 How many passes through collected data per update.
M (minibatch) 64 Minibatch size for each gradient step.

Why Did PPO Win?

The paper benchmarks PPO against TRPO, A2C, vanilla PG, and others on 7 MuJoCo robotics environments. PPO wins on almost all of them — not because it's the most theoretically rigorous, but because it hits the right balance:

TRPO: Theoretically motivated, hard constraint, 2nd order gradients → Works well, but complex and slow to implement PPO: Pragmatic approximation, simple clip, 1st order (Adam) only → Nearly same performance, 5x simpler code, 2x faster wallclock The paper's ε=0.2 clipping beats the KL-penalty approach in experiments. This is an empirical win, not a theoretical one — PPO is engineering that works remarkably well in practice.
Connecting to optimal control: PPO is doing approximate dynamic programming with neural networks. The critic is approximating the value function V*(s). The actor is approximating the optimal policy π*(s). The clipping is a practical stability constraint — the RL equivalent of a step-size bound in gradient descent.

Stable-Baselines3: A Trustworthy RL Toolbox

What it is, how it's structured, and why it matters for learning RL

What Is It?

Stable-Baselines3 (SB3) is a Python library that provides reliable, well-tested implementations of major deep RL algorithms. It was created because early RL code was notoriously hard to reproduce — two "implementations" of PPO could behave completely differently due to subtle differences in reward normalization, gradient clipping, or advantage computation.

SB3 is the successor to Stable-Baselines (which was itself a fork of OpenAI Baselines). It's written in PyTorch and is the current community standard for RL experimentation.

Algorithms Available

Algorithm Type Action Space When to Use
PPO On-policy, Actor-Critic Discrete + Continuous Default choice. Works well on almost everything.
A2C On-policy, Actor-Critic Discrete + Continuous Faster per update than PPO but less stable.
DQN Off-policy, Value-based Discrete only Classic; good for discrete action problems.
SAC Off-policy, Actor-Critic Continuous only State of the art for continuous control (robotics).
TD3 Off-policy, Actor-Critic Continuous only Very stable for continuous control; deterministic policy.
HER Off-policy wrapper Continuous Goal-conditioned tasks (robotics, manipulation).

The Design Philosophy

SB3 follows a consistent API across all algorithms. The core three things you always do are: 

from stable_baselines3 import PPO   # or A2C, DQN, SAC, TD3

# 1. CREATE: Wrap your environment with an algorithm
model = PPO("MlpPolicy", env, verbose=1)

# 2. TRAIN: Collect data and update the policy
model.learn(total_timesteps=50_000)

# 3. PREDICT: Use the trained policy
action, _states = model.predict(obs, deterministic=True)

This is the same three lines regardless of which algorithm you use. That's the point — SB3 abstracts away the implementation details so you can focus on understanding the algorithm conceptually and tuning hyperparameters.

What Happens Inside model.learn()?

For PPO specifically, here is what SB3 does under the hood when you call model.learn(total_timesteps=50_000): 

While total steps collected < 50,000:

  ┌── Data Collection Phase ─────────────────────────────┐
  │  Run current policy for n_steps=2048 timesteps       │
  │  Store: states, actions, rewards, dones, values, log_probs │
  │  Compute GAE advantage estimates                     │
  │  Normalize advantages (mean=0, std=1)                │
  └──────────────────────────────────────────────────────┘
        │
        ▼
  ┌── Optimization Phase ────────────────────────────────┐
  │  For each of n_epochs=10 epochs:                     │
  │    Shuffle the 2048 timesteps                        │
  │    For each minibatch of 64 samples:                 │
  │      Compute L_CLIP + L_VF + entropy                 │
  │      Adam gradient step                              │
  │      Clip gradient norm to 0.5                       │
  └──────────────────────────────────────────────────────┘

Key SB3-Specific Defaults (vs the Paper)

SB3's PPO defaults differ slightly from the paper's hyperparameters. It's worth knowing these:

Setting Paper (MuJoCo) SB3 Default Notes
n_steps 2048 2048 Same. Steps per rollout per environment.
batch_size 64 64 Same.
n_epochs 10 10 Same.
learning_rate 3e-4 3e-4 Same.
clip_range ε=0.2 0.2 Same.
gae_lambda 0.95 0.95 Same.
normalize_advantage not mentioned True SB3 normalizes advantages — important implementation detail!
max_grad_norm not mentioned 0.5 SB3 clips gradient norms for stability.

Useful SB3 Features You'll Use

Evaluation utility

from stable_baselines3.common.evaluation import evaluate_policy

mean_reward, std_reward = evaluate_policy(
    model, 
    eval_env, 
    n_eval_episodes=20,
    deterministic=True
)
print(f"Mean: {mean_reward:.1f} ± {std_reward:.1f}")

Save and load

model.save("my_ppo_cartpole")
loaded_model = PPO.load("my_ppo_cartpole", env=env)

Custom hyperparameters

model = PPO(
    "MlpPolicy",
    env,
    learning_rate=3e-4,
    n_steps=2048,
    batch_size=64,
    n_epochs=10,
    gamma=0.99,
    gae_lambda=0.95,
    clip_range=0.2,
    ent_coef=0.0,      # entropy coefficient c2
    vf_coef=0.5,       # value function coefficient c1
    verbose=1
)

Training callbacks — watching progress

from stable_baselines3.common.callbacks import EvalCallback

eval_callback = EvalCallback(
    eval_env,
    best_model_save_path="./best_model/",
    eval_freq=5000,
    n_eval_episodes=10,
    verbose=1
)
model.learn(total_timesteps=50_000, callback=eval_callback)
Why SB3 and not from scratch? At this stage of learning, your goal is to understand the algorithm, not debug implementation bugs. RL implementations have dozens of "implementation details" (advantage normalization, gradient clipping, value loss clipping, learning rate schedules) that can silently kill performance. SB3 gets all of these right, letting you focus on understanding what the hyperparameters do. After you understand PPO deeply, then write it from scratch.

MlpPolicy: The Neural Network Inside PPO

What the actual network looks like, how it parameterizes the policy and value function

What Does "MlpPolicy" Mean?

When you write PPO("MlpPolicy", env), you're telling SB3 to use a Multi-Layer Perceptron (a standard feedforward neural network) as the function approximator for both the actor and the critic.

SB3 offers three policy types:

Policy String Input Type Network When to Use
MlpPolicy Flat vector (1D) Dense layers (MLP) State is a vector: CartPole [4], MuJoCo [17+], etc.
CnnPolicy Images (pixels) Conv layers + MLP Atari games, visual observations
MultiInputPolicy Dict of inputs Mixed network Environments with multiple observation types

For CartPole, the observation is [cart_pos, cart_vel, theta, theta_dot] — a flat 4-dimensional vector. MlpPolicy is the correct choice.

The Exact Network Architecture

SB3's MlpPolicy for PPO uses this default architecture:

INPUT: state s = [x, ẋ, θ, θ̇] shape: (4,) │ ▼ ┌─────────────────────────────────────────────────────┐ │ SHARED FEATURE EXTRACTOR (optional) │ │ Identity for MlpPolicy — no shared layers │ │ (actor and critic have separate networks) │ └─────────────────────────────────────────────────────┘ │ │ ▼ ▼ ┌──────────────────┐ ┌──────────────────┐ │ ACTOR NETWORK │ │ CRITIC NETWORK │ │ │ │ │ │ Linear(4 → 64) │ │ Linear(4 → 64) │ │ Tanh │ │ Tanh │ │ Linear(64 → 64) │ │ Linear(64 → 64) │ │ Tanh │ │ Tanh │ │ Linear(64 → 2) │ │ Linear(64 → 1) │ │ Softmax │ │ (no activation) │ │ │ │ │ │ Output: │ │ Output: │ │ [P(left), │ │ V(s) — a scalar │ │ P(right)] │ │ value estimate │ └──────────────────┘ └──────────────────┘ │ │ ▼ ▼ sample action used to compute a ~ Categorical(P) advantage Â_t
Two layers of 64 units each, tanh activations. This is exactly what the PPO paper describes for the MuJoCo benchmarks: "a fully-connected MLP with two hidden layers of 64 units, and tanh nonlinearities."

Why Tanh and Not ReLU?

ReLU is more popular in supervised learning. RL tends to use Tanh for policy networks for a few reasons:

ReLU(x) = max(0, x) → Output range: [0, ∞) Tanh(x) = (e^x−e^-x)/(e^x+e^-x) → Output range: (−1, +1) In RL, policy activations often need to represent both positive and negative signals (lean left vs lean right, speed up vs slow down). Tanh is naturally centered at 0 and bounded, which keeps activations in a stable range across the many gradient updates during training.

How the Actor Outputs a Discrete Action

For CartPole (discrete actions: 0 or 1), the actor's final layer outputs 2 logits. These get converted to probabilities: 

logits = actor_network(state)      # shape: (2,)  e.g. [1.3, -0.2]

# Convert to probability distribution
probs = softmax(logits)            # e.g. [0.75, 0.25]

# During training: sample stochastically
action = Categorical(probs).sample()   # 0 or 1

# During evaluation (deterministic=True): take argmax
action = argmax(probs)             # 0 (higher probability)

The log-probability of the chosen action, log π(a|s), is stored during collection and used in the PPO clip objective. This is the log_prob in the training buffer.

How the Critic Outputs a Value

The critic network outputs a single scalar — V(s), the estimated total discounted future reward from this state: 

value = critic_network(state)      # shape: (1,)  e.g. [487.3]

# Interpretation: "From this state, I expect to accumulate
# roughly 487 more units of reward if I follow the current policy."

# CartPole max reward is 500 (500 timesteps × +1 each)
# So value ≈ 487 means "I'm almost certainly going to balance 
# until the time limit"

The Advantage Estimate Connecting Both

After collecting a rollout, SB3 computes the advantage for each timestep using the critic's value estimates and the actual rewards: 

# Pseudocode of what SB3 does after each rollout:

for t in reversed(range(T)):
    # One-step TD error (the "surprise")
    delta_t = rewards[t] + gamma * values[t+1] * (1 - dones[t]) - values[t]
    
    # GAE: exponential blend of TD errors
    advantages[t] = delta_t + gamma * gae_lambda * advantages[t+1] * (1 - dones[t])

# Normalize advantages across the batch
advantages = (advantages - advantages.mean()) / (advantages.std() + 1e-8)

That last line — normalization — is important and not in the original paper. It ensures advantages have zero mean and unit variance across the batch, which stabilizes training significantly.

Customizing the Network Architecture

You can change the architecture through the policy_kwargs argument: 

model = PPO(
    "MlpPolicy",
    env,
    policy_kwargs=dict(
        net_arch=[64, 64],          # default: two layers of 64
        activation_fn=torch.nn.Tanh # default: Tanh
    )
)

# Try a wider network:
policy_kwargs=dict(net_arch=[256, 256])

# Try a deeper network:
policy_kwargs=dict(net_arch=[64, 64, 64])

# Try separate arch for actor and critic:
policy_kwargs=dict(net_arch=dict(pi=[64, 64], vf=[64, 64]))
For CartPole, the default 2×64 is already overkill. CartPole is a nearly linear system — the optimal policy is approximately a linear function of the state. The PPO policy will converge to something close to your hand-crafted PD controller from Phase 3, but represented as weights in a neural network rather than explicit gains.

What the Network Actually Learns for CartPole

After training, if you inspect the learned policy, you'll find it behaves almost identically to your Phase 3 better_policy: 

# Your hand-crafted policy:
if theta + 0.1 * theta_dot > 0:
    action = 1   # push right
else:
    action = 0   # push left

# PPO's MlpPolicy learns something approximately equivalent to:
# action = 1  if  w1*theta + w2*theta_dot + w3*cart_pos + w4*cart_vel > 0
#
# The weights w1, w2, w3, w4 are encoded in the first layer of the network.
# The neural net doesn't "need" two layers for this problem —
# but it figures out the right linear combination regardless.

This is a key insight: for linear or near-linear systems, RL with neural networks rediscovers something close to what classical control theory would compute (an LQR controller). The difference is RL finds this without ever knowing the system dynamics — only from reward signals.

The big picture: PPO("MlpPolicy", env) is shorthand for: "Use two separate 2×64 tanh MLPs — one to represent π(a|s), one to represent V(s) — and optimize them together using the clipped surrogate objective with GAE advantage estimates, Adam optimizer, for K epochs per rollout." Every one of those words now means something to you.
